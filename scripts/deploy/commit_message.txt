Per-version summary
-------------------

v2.15.3 (2026-04-16) -- AI title category estimate yield
  * apps/buying/services/ai_title_category_estimate.py: drop the
    redundant `title_echo` verification (rows already match via
    `auction_id`); pad cached system block past Haiku 2048-token minimum
    so repeated batches benefit from cache_read pricing; few-shot
    vendor examples exclude rows where Mixed lots & uncategorized is
    >= 80%.
  * apps/buying/management/commands/estimate_auction_categories.py:
    add --missing-both (open/closing auctions with no AI mix and no
    manifest mix) with default 500 cap; --limit overrides.

v2.15.2 (2026-04-16) -- Retail-weighted manifest mix + AI Mixed-lot blend
  * apps/buying/services/valuation.py: `compute_and_save_manifest_
    distribution` now uses retail share per fast_cat_value (row-count
    fallback only when retail is null/zero); `_mix_for_auction` blends
    the "Mixed lots & uncategorized" bucket with AI title estimates
    when both exist so need_score SUMPRODUCT reflects known rows plus
    AI for the unmapped share.
  * `run_ai_estimate_for_swept_auctions`: remove 25-auction per-sweep
    cap; skip auctions that already have ai_category_estimates.
  * `recompute_auction_full`: refresh manifest distribution from
    ManifestRow before revenue / need_score rollup.
  * apps/buying/management/commands/recompute_buying_valuations.py:
    backfill path for retail-weighted mixes.

v2.15.1 (2026-04-16) -- Manifest pipeline optimizations + dev timelog
  * apps/buying/services/scraper.py: lazy singleton `requests.Session`
    (`_manifest_http_session`) for TLS connection reuse across
    paginated manifest GETs.
  * apps/buying/services/pipeline.py: preload CategoryStats, annotate
    queryset with `_has_manifest_rows` via Exists, bulk_create(batch_
    size=500), 1-deep ThreadPoolExecutor prefetch of the next
    auction's manifest.
  * apps/buying/services/manifest_dev_timelog.py (new): per-pull JSONL
    + time_summary.md when ENVIRONMENT=development.
  * apps/buying/management/commands/benchmark_manifest_pull.py (new):
    warm-up + baseline timing; flags --auction-id, --baseline-runs,
    --skip-warmup.
  * pull_manifests_budget / pull_manifests_nightly: inter-auction
    delay default 1.0s (was higher).
  * apps/buying/migrations/0012_category_stats_auction_lifecycle_item_
    indexes.py, 0015_manifest_pull_log_and_auction_manifest_pulled_at
    .py: supporting indexes + ManifestPullLog.
  * Baseline confirmed: ~38s / 1010 rows / ~26 rows/s (B-Stock
    10-items-per-page hard cap).

v2.15.0 (2026-04-15) -- Auction detail UX v3 (decision-flow layout)
  * frontend/src/pages/buying/AuctionDetailPage.tsx + decomposed
    cards (AuctionDetailsInfoCard, AuctionPrimaryCard,
    AuctionSecondaryCard, BuyingDetailSectionTitle).
  * New manifest / sweep UX: ManifestQueueDialog (Next up + Pull log),
    BuyingSweepProgressDialog, ManifestPullProgressPanel,
    useBuyingManifestPullProgress hook.
  * useLiveBuyingCountdown hook (per-row 1s interval only under
    threshold; no parent re-renders).
  * CategoryNeedDetail redesigned around raw 1-99 inputs + "sold
    items window since" date; CategoryNeedBars / NeedPill aligned.
  * Admin AssumptionsPage stub (staff-editable universal defaults
    placeholder).
  * Supporting utils: valuationParse.ts, buyingOptimisticCache.ts.
  * .ai/extended/ux-spec.md (new): authoritative UI/UX design spec.

v2.14.1 (2026-04-15) -- SOCKS5 hardened for all B-Stock HTTP
  * apps/buying/services/scraper.py: route every *.bstock.com request
    (not just search) through SOCKS5 when enabled; PIA-friendly
    knobs (socks5:// vs socks5h://, resolved-IP override via
    BUYING_SOCKS5_PROXY_IP, BUYING_SOCKS5_LOCAL_DNS, dev audit via
    BUYING_SOCKS5_DEV_AUDIT).
  * workspace/tests/socks5_egress_probe.py (new): 6-step diagnostic
    (resolve host -> direct -> socks5+host -> socks5+IP ->
    socks5h+host -> optional bstock probe).
  * .ai/extended/vpn-socks5.md (new): PIA setup, env keys, IP
    rotation, troubleshooting.

v2.14.0 (2026-04-15) -- Item cost (est_shrink) + need_score_1to99
  * apps/inventory/models.py: PurchaseOrder.est_shrink default 0.15;
    Item.cost derived from PO retail + est_shrink + listing retail;
    Item.cost refreshes in Django save() when PO/line retail or PO
    assignment changes.
  * apps/inventory/migrations/0022_category_stats_auction_lifecycle
    _item_indexes.py, 0023_po_est_shrink_remove_cost_pipeline_fields
    .py (DESTRUCTIVE -- drops shrinkage_rate/misfit_rate/avg_sell_
    through/avg_fulfillment/shrink_retail_est/mistracked_retail/
    misfit_sales_amt).
  * apps/buying/services/category_stats_sql.py (new),
    category_need.py: SQL aggregate refresh + 1-99 scaling.
  * apps/buying/services/valuation.py: auction need_score / priority
    = SUMPRODUCT(CategoryStats.need_score_1to99 x manifest/AI mix).
  * apps/buying/management/commands/compute_daily_category_stats.py
    (new): Heroku Scheduler daily job (SQL refresh + open-auction
    recompute + category-need cache invalidation).
  * apps/inventory/management/commands/recompute_all_item_costs.py
    (new): one-shot backfill.
  * Deleted legacy: compute_vendor_metrics, compute_po_cost_analysis,
    compute_item_cost, recompute_cost_pipeline.
