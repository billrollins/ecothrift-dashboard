Processing UI Comparison: M3 vs M5

Visual helper to identify fundamental processing differences before implementation.

M3Universal Items + Smart Batch

  • All manifest units become Item rows up front.
  • Processor mostly performs bulk updates over existing Item rows.
  • No unitization action exists in core flow.

M5Hybrid Policy-Driven Unitization

  • Units start as StockLot quantity, not Item rows.
  • Processor unitizes only needed quantity for shelf.
  • Immediate itemization is policy/override-driven for exceptions.
Differentiator M3 Processing Experience M5 Processing Experience Impact
Inventory shown on processing screen Mostly Item queues and BatchGroups Dual queues: Deferred Lots + Immediate Items M5 adds explicit reserve flow visibility
When Item rows are created Before processing (near 100% at intake) During processing via unitize action M5 lowers early row growth and front-loaded labor
Processor primary action Apply attributes to existing items in bulk Decide unitize quantity then create sellable items M5 gives tighter control over shelf replenishment
Policy visibility Minimal (batch heuristics) Explicit policy reason shown per lot/item M5 requires policy governance but improves explainability
Speed for very large manifests Good, but still writes all item rows early Very strong, defers item row creation M5 scales better with repeated commodity units
Mental model simplicity Simpler: everything is an item More complex: lots + items + policy M3 easier training, M5 stronger operational control
